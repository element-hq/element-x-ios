## 1. ЛИЦЕНЗИРОВАНИЕ — КРИТИЧЕСКИЙ ВОПРОС

> **Это самый важный пункт всего разговора. Без решения этого вопроса публикация в App Store невозможна.**

### 1.1. Что такое AGPL v3 и почему это проблема

#### Контекст для вас (разработчика)

Element X iOS распространяется под лицензией **GNU Affero General Public License v3 (AGPL v3)** — это самая строгая из широко используемых open-source лицензий. Она требует:

1. **Открытие исходного кода.** Любой, кто получает программу (включая через сеть), имеет право получить полный исходный код, включая все модификации.
2. **Сохранение лицензии.** Все производные работы (наш форк) автоматически подпадают под AGPL v3. Мы не можем сменить лицензию.
3. **Право на модификацию и распространение.** Пользователи должны иметь право свободно изменять и распространять программу.

**Почему это конфликтует с App Store:**

Apple App Store имеет собственные Условия использования (Terms of Service), которые:
- **Ограничивают установку** — приложение можно установить только на ограниченное количество устройств, привязанных к Apple ID
- **Контролируют распространение** — приложения распространяются ТОЛЬКО через App Store, пользователь не может свободно передать .ipa-файл другому человеку
- **Накладывают DRM** — App Store применяет FairPlay DRM (защиту от копирования)

Эти ограничения Apple **прямо противоречат** правам, которые AGPL v3 гарантирует пользователям. Другими словами:
- AGPL говорит: «Пользователь имеет право свободно копировать и распространять программу»
- Apple говорит: «Распространение только через наш магазин, с нашей защитой»

Это **юридический конфликт**, и он означает, что простая публикация AGPL-приложения в App Store создаёт правовой риск.

#### Как объяснить заказчику

> «Element X iOS — это open-source проект, который распространяется под лицензией AGPL v3. Эта лицензия требует, чтобы пользователи имели полную свободу копировать и распространять приложение. Но правила App Store это запрещают — Apple контролирует, как приложение устанавливается и распространяется. Получается юридическое противоречие: лицензия разрешает одно, а правила магазина — другое.
>
> Это не теоретическая проблема. Компания Element (авторы Element X) является правообладателем и может предъявить претензии. Apple также может снять приложение с публикации, если обнаружит нарушение.
>
> Для легальной публикации нам нужно получить от Element **коммерческую лицензию** — это отдельное соглашение, которое даёт нам право распространять приложение через App Store без ограничений AGPL.»

## 2. PUSH-УВЕДОМЛЕНИЯ: FCM уже реализован

### Контекст для вас

В ТЗ написано: «Push-уведомления через Firebase Cloud Messaging (FCM)». Оригинальный Element X iOS использовал только APNs, но мы уже **реализовали поддержку FCM** согласно ТЗ:

- Firebase SDK (v11.8.x) добавлен в проект через SPM
- Создан `FirebaseNotificationService` для управления FCM-токенами
- Приложение автоматически регистрирует pusher с FCM-токеном на Matrix-сервере
- Код покрыт 14 unit-тестами, все проходят

**Решение по APNs vs FCM уже не требуется.** FCM реализован. Осталось только получить конфигурацию от заказчика.

### Как объяснить заказчику

> «Согласно вашему техническому заданию, мы реализовали push-уведомления через Firebase Cloud Messaging. Код готов и протестирован. Для запуска push-уведомлений нам нужны от вас три вещи:
>
> 1. Файл `GoogleService-Info.plist` — его можно скачать в Firebase Console вашего проекта
> 2. Настроить Sygnal для FCM — добавить FCM-конфигурацию в ваш Sygnal push-gateway
> 3. Загрузить APNs-ключ в Firebase Console — мы сгенерируем ключ, вам нужно добавить его в настройки Cloud Messaging»

### Вопросы к заказчику

1. **«У вас есть Firebase-проект? Если да, можете ли вы скачать `GoogleService-Info.plist` из Firebase Console?»**
   - Если нет Firebase-проекта — его нужно создать (бесплатно, ~10 минут)

2. **«Ваш Sygnal push-gateway уже настроен для FCM?»**
   - URL Sygnal'а?
   - Если Sygnal настроен только для APNs — нужно добавить FCM-конфигурацию

3. **«Готовы ли вы загрузить APNs-ключ в Firebase Console?»**
   - Мы сгенерируем ключ (файл `.p8`) и предоставим вам: Key ID, Team ID
   - Вам нужно добавить его в Firebase Console → Project Settings → Cloud Messaging

### Рекомендация

## 3. ЗВОНКИ: Element Call vs. Jitsi

### Контекст для вас

В ТЗ упоминается **Jitsi** для групповых звонков. Но Element X iOS использует **Element Call** — это совершенно другая технология:

| | Jitsi | Element Call |
|---|-------|-------------|
| **Протокол** | Собственный (через WebRTC) | MatrixRTC (через протокол Matrix) |
| **Медиа-сервер** | Jitsi Videobridge (JVB) | LiveKit SFU |
| **Интеграция в Element X** | Отсутствует | Встроена |
| **Код в приложении** | Нет | Есть |

Переделка Element X на Jitsi — это фактически **разработка новой функции**, что выходит за рамки форка и добавит десятки часов.

### Как объяснить заказчику

> «В техническом задании упоминается Jitsi для видеозвонков. Element X iOS использует другую систему — Element Call с сервером LiveKit. Это современная замена Jitsi, которая глубже интегрирована в протокол Matrix.
>
> Хорошая новость: звонки 1:1 и групповые уже работают в Element X. Но для этого на вашей стороне нужен сервер LiveKit (вместо или в дополнение к Jitsi).
>
> Если у вас есть только Jitsi и нет возможности развернуть LiveKit — звонки в приложении работать не будут. Переделка на Jitsi — это отдельная большая задача, которая не входит в текущий проект.»

### Вопросы к заказчику

1. **«Какая инфраструктура для звонков у вас развёрнута?»**
   - Jitsi Meet?
   - LiveKit SFU?
   - Element Call?
   - Ничего (звонки не настраивались)?

2. **«Если у вас только Jitsi — готовы ли вы развернуть LiveKit SFU и Element Call?»**
   - LiveKit — это open-source, можно развернуть самостоятельно
   - Element также предоставляет hosted-решения

3. **«Насколько критичны групповые звонки для первой версии?»**
   - Если не критичны — можно отложить звонки до v2 и сосредоточиться на сообщениях
   - Звонки 1:1 через Matrix VoIP могут работать и без Element Call (через прямое WebRTC-соединение)

4. **«Есть ли у вас TURN/STUN-серверы?»**
   - Без них звонки будут работать только в одной локальной сети, но не через мобильный интернет

### Рекомендация

> Если заказчик может развернуть LiveKit — используем Element Call (без изменений в коде). Если не может — обсудить отключение звонков в v1 или отложить до решения инфраструктурного вопроса.

## 4. МИНИМАЛЬНАЯ ВЕРСИЯ iOS

### Контекст для вас

ТЗ указывает **iOS 16+**. Наш форк Element X iOS требует минимум **iOS 18.0** — кодовая база использует API, доступные только начиная с iOS 18.

Поддержка iOS 16 или iOS 17 **невозможна** без серьёзной переработки кода, что не входит в рамки форка.

### Как объяснить заказчику

> «В техническом задании указана поддержка iOS 16 и выше. Element X iOS технически не может работать на версиях ниже iOS 18 — код использует системные функции, доступные только начиная с этой версии.
>
> На практике это означает: iPhone XS и новее с обновлённой iOS. Это подавляющее большинство активных устройств.
>
> Обойти это ограничение без полной переработки кода невозможно — это ограничение самого проекта Element X.»

### Вопросы к заказчику

1. **«Какая минимальная версия iOS у ваших пользователей?»**
   - Если есть статистика — отлично. Если нет — iOS 18+ покрывает подавляющее большинство.

2. **«Принимаете ли вы ограничение iOS 18+ вместо iOS 16+?»**
   - Это не наш выбор — это ограничение исходного проекта Element X.

### Рекомендация

> **iOS 18.0+** — это единственный поддерживаемый вариант для текущей кодовой базы Element X.

## 5. СЕРВЕРНАЯ ИНФРАСТРУКТУРА

### Контекст для вас

Приложение бесполезно без работающего бэкенда. Нам нужно знать, что готово, а что нет, чтобы правильно спланировать интеграцию.

### Вопросы к заказчику

#### 5.1. Matrix Homeserver

1. **«Ваш Matrix homeserver работает и доступен?»**
   - URL homeserver'а?
   - Версия Synapse (или другой реализации)?
   - Sliding Sync настроен? (Element X требует Sliding Sync proxy или нативную поддержку)

2. **«Homeserver доступен из интернета или только из внутренней сети?»**
   - Для разработки и тестирования мне нужен доступ извне
   - Для App Store Review — тоже (модераторы Apple будут тестировать)

3. **«Можете ли вы предоставить тестовые аккаунты для разработки?»**
   - Минимум 2 аккаунта для тестирования сообщений и звонков

#### 5.2. Аутентификация (OIDC)

4. **«Как настроена аутентификация на вашем сервере?»**
   - Стандартный логин/пароль Matrix?
   - OIDC / SSO (OpenID Connect)?
   - Если OIDC — какой провайдер? (Keycloak, Auth0, собственный?)

5. **«Если OIDC — готовы ли вы зарегистрировать наше приложение как OIDC-клиент?»**
   - Нам понадобится: Client ID, Redirect URI
   - Мы предоставим точный Redirect URI после настройки bundle ID

6. **«Настроен ли файл `.well-known/matrix/client` на вашем домене?»**
   - Это нужно для автоматического определения homeserver'а по доменному имени

#### 5.3. Push-уведомления (Sygnal + FCM)

7. **«Sygnal push-gateway развёрнут и работает?»**
   - URL Sygnal'а?
   - Настроен ли для FCM?

8. **«Можете ли вы предоставить `GoogleService-Info.plist` из вашего Firebase-проекта?»**
   - Скачать в Firebase Console → Project Settings → General → iOS app
   - Если Firebase-проекта нет — его нужно создать (бесплатно)

9. **«Готовы ли вы загрузить APNs-ключ в Firebase Console?»**
   - Мы сгенерируем APNs-ключ (файл `.p8`, Key ID, Team ID)
   - Вам нужно добавить его в Firebase Console → Cloud Messaging

#### 5.4. Звонки

10. **«Развёрнуты ли у вас серверы для звонков?»**
    - LiveKit SFU? URL?
    - Element Call? URL развёртывания?
    - TURN/STUN-серверы? URL и учётные данные?
    - Или только Jitsi?

#### 5.5. Общие вопросы

11. **«Когда вся инфраструктура будет готова к тестированию?»**
    - Нам она нужна к концу 2-й недели разработки (День 9–10)
    - Если не будет готова — разработка встанет

12. **«Есть ли у вас staging/тестовая среда, отдельная от продакшна?»**
    - Для разработки лучше использовать отдельную среду, чтобы не затрагивать реальных пользователей
