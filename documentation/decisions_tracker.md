### D-001: Лицензирование AGPL v3 / App Store

| Поле | Значение |
|------|----------|
| **Статус** | 🔴 Открыт |
| **Критичность** | 🚨 Блокер |
| **Владелец** | Заказчик + Разработчик |
| **Дедлайн решения** | День 20 (решение о подходе), День 27 (лицензия подписана) |
| **Фазы, которые блокирует** | Фаза 9 (публикация в App Store) |
| **Дата создания** | 2026-02-08 |
| **Дата последнего обновления** | 2026-02-08 |

**Описание проблемы:**
Element X iOS лицензирован под AGPL v3. Условия AGPL v3 юридически конфликтуют с Terms of Service Apple App Store (Apple накладывает DRM и ограничивает распространение, AGPL требует свободного распространения). Публикация AGPL-приложения в App Store без коммерческой лицензии создаёт юридический риск.

**Варианты решения:**

| Вариант | Описание | Рекомендация |
|---------|----------|-------------|
| A. Коммерческая лицензия от Element | Связаться с Element (New Vector Ltd), получить отдельную лицензию | ✅ Рекомендуется |
| B. Открыть исходный код форка | Опубликовать репозиторий с изменениями | ⚠️ Не решает конфликт с App Store |
| C. Использовать старый Element iOS (Apache 2.0) | Сменить базовый проект на устаревший, но легальный | ⚠️ Запасной вариант |
| D. Принять юридический риск | Публиковать без лицензии | ⚠️ Риск |

**Контрольные точки:**

| Дата/День | Контрольная точка | Статус |
|-----------|-------------------|--------|
| День 1 | Запрос отправлен в Element | ⬜ Не выполнено |
| День 5 | Проверка ответа, повторный запрос если нет | ⬜ Не выполнено |
| День 11 | Ответ получен или эскалация | ⬜ Не выполнено |
| День 20 | Решение о подходе принято | ⬜ Не выполнено |
| День 27 | Лицензия подписана | ⬜ Не выполнено |

**Решение:** *(будет заполнено после принятия решения)*

**Примечания:**
- Запрос нужно отправить немедленно — ответ может занять 1–3 недели
- Разработка может начаться параллельно (Фазы 1–8 не требуют лицензии)
- Публикация в App Store (Фаза 9) невозможна без решения

### D-002: Push-уведомления — APNs или FCM

| Поле | Значение |
|------|----------|
| **Статус** | 🟡 В процессе |
| **Критичность** | 🚨 Блокер |
| **Владелец** | Заказчик (предоставляет GoogleService-Info.plist и настройку Sygnal) |
| **Дедлайн решения** | До начала тестирования push-уведомлений |
| **Фазы, которые блокирует** | Фаза 5 (тестирование Push-уведомлений) |
| **Дата создания** | 2026-02-08 |
| **Дата последнего обновления** | 2026-02-10 |

**Описание проблемы:**
ТЗ указывает FCM (Firebase Cloud Messaging). Инфраструктура FCM добавлена в приложение (Firebase SDK, условный выбор APNs/FCM, заглушка GoogleService-Info.plist). Осталось: реальные значения Firebase-проекта от заказчика и настройка Sygnal для FCM на стороне сервера.

**Что сделано:**
- Firebase SDK (v11.8.x) добавлен через SPM
- `FirebaseNotificationService` — инициализация Firebase, управление FCM-токеном
- Настройка `PushProvider` в `AppSettings` (по умолчанию `.firebase`)
- Условная логика в `AppCoordinator` — FCM или APNs в зависимости от настройки
- `NotificationManager` — регистрация pusher с FCM-токеном
- Заглушка `GoogleService-Info.plist` с TODO-значениями

**Что осталось (требует заказчика):**
- Реальные значения `GoogleService-Info.plist` (из Firebase Console)
- Настройка Sygnal для FCM (серверная сторона)
- Загрузка APNs-ключа в Firebase Console
- Сквозное тестирование push-уведомлений

**Решение:** FCM выбран согласно ТЗ. Инфраструктура готова, ожидает конфигурацию от заказчика.

ТЗ упоминает Jitsi для звонков. Element X iOS использует Element Call (MatrixRTC + LiveKit SFU). Jitsi не поддерживается. Если у заказчика нет LiveKit — звонки работать не будут.

**Варианты решения:**

| B. Отключить звонки в v1 | Нет | Минимальные (скрыть UI) |
| C. Интеграция Jitsi | Только Jitsi-сервер | Значительные (+20–40ч, вне рамок проекта) |
